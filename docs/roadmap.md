# Quiver — Roadmap & Definitions of Done

Quiver is built phase by phase. A phase is not "done" until its Definition of Done (DoD) is met, tested, documented, and merged. Each phase boundary opens a release PR `develop` → `main` and tags a SemVer release.

> For a current capability / coherence / security / readiness snapshot and the gap matrix the
> post-launch phases close against, see [State of Quiver](analysis/state-of-quiver-v0.17.md).

## Gates

- **Plan gate** — the owner approves the build plan before any work. *(passed)*
- **Design gate (end of Phase 0)** — the owner approves this design package (PR-A…D) **before any implementation code is written.**
- Thereafter, each phase ships behind its DoD and a tagged release.

## Phase 0 — Design (no code) → docs only

**Goal:** a complete, reviewed technical design so implementation is de-risked on paper.

**Deliverables:** architecture + C4; on-disk format spec; index design (HNSW/Vamana/IVF + quantization) with cited papers; concurrency model; security threat model + crypto choices; API & wire-protocol spec; benchmark methodology; risk register; this roadmap; the repo-scaffold plan; ADRs for every major decision.

**DoD:** the owner has reviewed and approved the design package; the riskiest unknown — disk-resident index memory/recall behavior — has an evidence-based plan (analytical budget + a spike plan). No SemVer tag (design only).

## Phase 1 — Single-node core (usable) → `v0.1.0`

**Goal:** one binary, end-to-end: encrypted collection → upsert → filtered k-NN → live TUI.

**Scope:** Cargo workspace scaffold + `justfile` + manual CI + Dockerfile + README/CONTRIBUTING/SECURITY; storage engine (segments + WAL + crash recovery + checksums); HNSW; SIMD kernels (f32 + int8 with feature detection); REST + gRPC; basic payload filtering; encryption-at-rest secure-by-default; TLS via `rustls`; TUI MVP (live metrics + collection browser); benchmark harness + first SIFT1M numbers; Python SDK (uv); seeded demo.

**DoD (testable):**
- `quiver serve` + `quiver tui` run from one binary; `just demo` seeds an encrypted collection.
- Over **both** REST and gRPC and the Python SDK: create collection, upsert vectors with payloads, run top-k with a metadata filter, get correct results.
- Encryption-at-rest is on by default; data files are ciphertext on disk (verified by a test).
- A process **kill mid-write** recovers with no corruption and no lost acknowledged writes (crash-recovery test passes).
- Recall@10 on SIFT1M ≥ a documented threshold at a documented `ef`; numbers recorded in `docs/benchmarks/`.
- Coverage gate ≥ 70%; `just verify` green.

## Phase 2 — Memory frugality → `v0.2.0`

**Goal:** large datasets served from a small RAM budget, proven against competitors.

**Scope:** disk-resident graph (DiskANN/Vamana) + IVF; quantization (PQ/scalar/binary + hamming pre-filter + exact re-rank); recall/latency/memory knobs per collection; hybrid search (vector + filter + optional BM25); 10M+ disk-path benchmarks vs Qdrant + LanceDB; TypeScript SDK; MCP server; LangChain/LlamaIndex adapters.

**DoD:** on a 10M+ dataset, Quiver serves a documented recall@10 using a **fraction of the RAM** of Qdrant/LanceDB on identical hardware, with a reproducible methodology and raw numbers published. Quantization knobs documented with a tradeoff table. Coverage ≥ 75%.

**Status (on `develop`):** the disk-resident graph + IVF, quantization, per-collection index/quant knobs, the storage-engine rewrite (row-addressed segments, roaring tombstones, compaction, secondary indexes), **hybrid filtered search** (a selectivity planner that pre-filters to an exact scan or post-filters ANN, reachable over REST/gRPC, the MCP server, and the Python/TypeScript SDKs), and the LangChain + LlamaIndex adapters are all in and tested. The quantization/index tradeoff table and the disk-path recall/compression figures are recorded; the remaining DoD item is the published **10M reference-hardware** head-to-head (recall@10 vs RAM, against Qdrant/LanceDB) — its methodology and runbook are written, but the raw RSS/QPS require dedicated hardware and are never produced on the shared dev box or fabricated. `v0.2.0` is gated on those numbers plus owner approval.

## Phase 3 — Security depth + cockpit polish → `v0.3.0`

**Scope:** client-side payload encryption with a documented trust boundary; RBAC + scoped API keys + optional mTLS; audit logging; crypto-shredding; secret/KMS handling; the TUI **constellation view** (2D projection + nearest-neighbor highlight + interactive query); security testing (fuzz the protocol + on-disk format, `cargo audit`/`deny`, threat-model verification).

**DoD:** a payload upserted with client-side encryption is provably unreadable by the server (test); RBAC denies cross-tenant/over-scope access (tests); fuzz targets run clean for a documented duration; the cockpit demo (asciinema cast) is recorded. Coverage ≥ 80%.

**Status (shipped on `develop` for the `v0.3.0` tag):** all five slices are in and tested — client-side payload encryption (cross-language KATs); RBAC + scoped keys + optional mTLS; an append-only audit log; per-collection-DEK envelope encryption with **crypto-shredding**; `QUIVER_MASTER_KEY_FILE` secret handling; the cockpit **constellation view** (random-projection scatter + nearest-neighbour highlight + interactive re-query); and `cargo-fuzz` targets for the wire and on-disk parsers (clean over a bounded run). Core-engine coverage is **93% line**. The only artifact produced off the shared dev box is the cockpit's asciinema cast — `scripts/record-cockpit-cast.sh` records it on a real terminal. Note: the per-collection envelope is a pre-1.0 at-rest format change, so a `v0.2.0` encrypted store must be re-created.

## Phase 4 — Advanced / stretch features → `v0.4.0`, `v0.5.0`, … (launch is `v1.0.0`, several releases out)

Unlike the earlier phases, Phase 4 is a **backlog shipped incrementally**: each minor release (`v0.4.0`, `v0.5.0`, …) delivers a coherent, owner-gated subset, and **`v1.0.0` is reserved for the launch release** once the backlog and the launch polish below are complete. Releases ship incrementally — **`v0.4.0`**, **`v0.5.0`**, **`v0.6.0`**, **`v0.7.0`**, **`v0.8.0`**, **`v0.9.0`**, **`v0.10.0`**, **`v0.11.0`**, **`v0.12.0`**, **`v0.13.0`**, **`v0.14.0`**, **`v0.15.0`**, and **`v0.16.0`** are out; we remain deliberately far from `v1.0.0`.

**Backlog (rough priority):** incremental in-place updates (SpFresh-style); migration importers (Qdrant/Chroma/pgvector); multi-vector / late-interaction scoring; optional leader-follower replication (clearly labeled); the **experimental** DCPE feature flag (published scheme only, honest caveats); then the launch polish — docs site, benchmark-table fill-in, regenerated TUI cast, published load-test results.

**`v0.4.0` ships two backlog items:** **incremental in-place index updates** — SpFresh/LIRE for **IVF**, maintained in memory so the `kill -9` crash gate is untouched by construction ([ADR-0023](adr/0023-incremental-in-place-updates.md); a durable on-disk incremental index and graph (FreshDiskANN) updates are sequenced as later increments) — and **migration importers**, `quiver admin import` loading Qdrant/Chroma/pgvector exports into Quiver collections with filterable fields ([ADR-0024](adr/0024-migration-importers.md); see [`migration.md`](migration.md)).

**`v0.5.0` ships** **HNSW incremental delete** — O(1) soft-delete with search-time tombstone filtering and an amortized rebuild ([ADR-0026](adr/0026-hnsw-incremental-delete.md)); a **neighbor-bounded IVF reassignment** that brings LIRE rebalancing to its `O(nlist + |list|)` target ([ADR-0023](adr/0023-incremental-in-place-updates.md)); a **unified secure database-open path** shared by the server, the MCP server, and the CLI so a data directory is portable between them; and the **design** for a durable on-disk incremental index ([ADR-0025](adr/0025-durable-incremental-index.md)).

**`v0.6.0` ships** the **durable on-disk incremental IVF index** — the index is snapshotted at checkpoint, referenced by the manifest under the same atomic swap as the segments, and recovered on open by loading the snapshot and replaying the WAL tail instead of an `O(N)` rebuild, with the `kill -9` crash gate extended to cover it ([ADR-0025](adr/0025-durable-incremental-index.md), Accepted) — and a **live Qdrant migration connector**: `quiver admin import --qdrant-url` pulls a running collection directly over HTTP ([ADR-0027](adr/0027-live-migration-connectors.md)). Live Chroma/Postgres connectors and leader-follower replication are sequenced for `v0.8.0`+.

**`v0.7.0` ships** **multi-vector / late-interaction (ColBERT) retrieval** — a collection can be created `multivector`, storing each document as a group of token-vector rows over the existing row-addressed store (no on-disk format change, so the `kill -9` crash gate is untouched) and ranking documents by **MaxSim** late interaction: nearest-neighbour candidate generation over the token pool, then an exact MaxSim re-rank with an optional payload filter. The token pool compresses under the same IVF+PQ / disk path, so ColBERT's storage cost showcases the memory-frugality wedge. Reachable from the embeddable database, REST + gRPC, the MCP server, and the Python/TypeScript SDKs ([ADR-0028](adr/0028-multi-vector-late-interaction.md)). Live Chroma/Postgres connectors move to `v0.8.0`.

**`v0.8.0` ships** the **live Chroma and Postgres migration connectors**, completing one-command live migration from all three supported sources: `quiver admin import --chroma-url` pulls a running Chroma collection over its v2 HTTP API (resolving the collection name to an id, then paginating `get`), and `--postgres-url` pulls pgvector rows from a running Postgres via `row_to_json` — both reusing the offline importer's normalization and write path ([ADR-0029](adr/0029-live-chroma-postgres-connectors.md)). Chroma adds no new dependency (it reuses the `ureq` seam); Postgres adds the blocking `postgres` driver with rustls/`ring` TLS, leaving the embeddable engine tokio-free. Leader-follower replication and the experimental DCPE flag are sequenced for later increments.

**`v0.9.0` ships** **asynchronous leader-follower replication** — a follower applies the leader's committed operation log (the WAL's own ops) and serves reads, lagging by the replication delay. The leader exposes an admin-scoped `Replicate` gRPC stream that ships a logical snapshot, then the live commit tail (the handler subscribes inside the same engine critical section it snapshots, so the stream is race-free with no dedup); a follower (`QUIVER_LEADER_URL`) applies the stream, serves reads, and **refuses writes**. It reuses the WAL op and the recovery apply path, so the on-disk format and the `kill -9` crash gate are untouched. Synchronous replication, failover, and consensus are explicit non-goals — single-node stays the primary topology, and this is a clearly-labelled advanced feature ([ADR-0030](adr/0030-leader-follower-replication.md); see [`replication.md`](replication.md)). The experimental DCPE flag, graph FreshDiskANN incremental updates, and the multi-vector follow-ups are sequenced for later increments.

**`v0.10.0` ships** the **experimental DCPE vector-encryption mode** — opt-in, per-collection, off by default. A client encrypts embeddings with the published **Scale-And-Perturb** distance-comparison-preserving scheme (Fuchsbauer et al., ePrint 2021/1666; built only from audited RustCrypto primitives — ChaCha20, HMAC-SHA256, HKDF-SHA256), so an untrusted server can answer approximate nearest-neighbour queries over the ciphertexts **without ever holding the plaintext vectors or the key**. It is deliberately honest about its limits: **L2-only**, **not** semantically secure, and leaking the approximate distance-comparison relation **by design** (it composes with, but does not replace, encryption-at-rest). Shipped as a `quiver_crypto::dcpe` reference cipher, a `vector_encryption` collection flag across REST/gRPC/MCP/SDKs, a native Python `DcpeCipher` (validated against the Rust reference by a cross-language known-answer test), and an end-to-end gate proof that the plaintext vector never reaches disk ([ADR-0031](adr/0031-dcpe-vector-encryption.md); see [`security/dcpe.md`](security/dcpe.md)). Graph FreshDiskANN incremental updates, the multi-vector follow-ups, a native TypeScript DCPE cipher, and the documented DCPE hardening (component shuffle, distribution normalisation) are sequenced for later increments.

**`v0.11.0` ships** a **semantically secure client-side vector-encryption mode** — the IND-CPA complement to DCPE, opt-in and off by default. With `vector_encryption = client_side`, a client seals each vector with XChaCha20-Poly1305 (the audited AEAD already used for at-rest and payloads — no new cryptography) and stores the ciphertext blob plus a zero placeholder vector; the server learns **nothing** about the vectors (no coordinates, distances, or geometry) and does **no** ranking. The honest trade-off, stated plainly: since the server can't rank opaque ciphertext, retrieval is a **fetch** (an optional cleartext payload filter narrows the set) and the client decrypts and ranks locally — so it suits small/medium or pre-filtered collections, not unfiltered top-k at 10M scale. There is no on-disk format change and the `kill -9` crash gate is untouched. Shipped as a `quiver_crypto::vector` reference cipher, the mode enforced server-side (no index, ranked search rejected) with a `fetch` path across REST/gRPC/MCP, native Python and TypeScript ciphers with a **bit-exact** cross-language known-answer test, client-side `search` helpers, and an end-to-end gate proof ([ADR-0032](adr/0032-client-side-vector-encryption.md); see [`security/client-side-vectors.md`](security/client-side-vectors.md)). The flag migrated from `encrypted_vectors` to a three-valued `vector_encryption` enum (byte-compatible on disk). Graph FreshDiskANN incremental updates, the multi-vector follow-ups, a native TypeScript DCPE cipher, and the DCPE hardening steps remain sequenced for later increments.

**`v0.12.0`** is a documentation & packaging release (no functional change): it corrects the install instructions — the `quiver-cli` name on crates.io is an **unrelated third-party crate**, so the `quiver` binary is installed from source (`cargo install --path crates/quiver-cli`) and the SDKs from `sdks/` (publishing to crates.io/PyPI/npm is a launch-time task) — replaces the dynamic GitHub-API README badges (license/release/stars) with static ones that render everywhere, and repairs UTF-8 mojibake in the status banner.

**`v0.13.0` ships** **graph-index incremental updates** — the last index family that still rebuilt on every write. The Vamana and disk-resident DiskVamana graphs now use **FreshDiskANN's StreamingMerge** model (Singh et al. 2021): the batch-built graph is a read-only base, recent inserts land in a small in-memory **delta graph** searched alongside it, and deletes are `O(1)` **tombstones** filtered at query time with live-fraction beam widening; a query merges base and delta candidates by the metric's ordering. Consolidation is a **derived rebuild** from the store once the pending work (delta + tombstones) passes a 20% churn threshold — so graph updates become size-independent while the index stays in memory and derived, the on-disk artifact keeps its write-once contract, and the `kill -9` crash gate is untouched by construction (matching the IVF and HNSW incremental work, ADR-0023/0026; distinct from the durable IVF index of ADR-0025). An in-place on-disk StreamingMerge and delta persistence are sequenced as later, separately-gated increments ([ADR-0033](adr/0033-graph-incremental-freshdiskann.md)). The multi-vector follow-ups, a native TypeScript DCPE cipher, and the DCPE hardening steps remain sequenced for later increments.

**`v0.14.0` ships** the **multi-vector / ColBERT follow-ups** deferred by ADR-0028, in two of three parts ([ADR-0034](adr/0034-multivector-followups.md)). **(A) Incremental multi-vector index maintenance** — document upsert/delete now dispatch to the underlying ANN index incrementally (the shared per-point insert/delete path), instead of marking the collection `stale` for a full token-pool rebuild, so a document write is size-independent and the last write-then-rebuild path is gone. **(C) ColBERTv2 residual compression + PLAID pruning** — an opt-in `colbert` index for `multivector` collections holds coarse `kmeans` centroids + per-token *(centroid id, PQ residual code)* in RAM (residuals trained under L2 for faithful reconstruction; the collection metric applies at scoring) and prunes candidate generation by scoring centroids first, with the exact token vectors staying on the encrypted store for the MaxSim re-rank — the memory-frugal late-interaction story, creatable across REST/gRPC/MCP and the SDKs. Both parts stay in memory and derived (rebuilt from the store on open), so there is **no on-disk format change and the `kill -9` crash gate is untouched by construction**. **(B) Native variable-stride document rows is deferred**, honestly: its payoff is storage locality / id-space only (candidate generation still needs a token-pool index, so it removes no work), and ADR-0034 gates it on a *measured* locality benefit that can only be produced on owner reference hardware — so the on-disk change is not shipped on faith and is revisited in a later increment. A native TypeScript DCPE cipher and the DCPE hardening steps are taken up in `v0.15.0`.

**`v0.15.0` ships** the **documentation and DCPE polish before launch** ([ADR-0035](adr/0035-docs-site-and-dcpe-hardening.md)). An **mdBook documentation site** under `apps/docs` (pure Rust, no Node toolchain) ties the canonical docs into a reader's journey — concepts, quickstart, self-hosting, the feature guides, the REST/gRPC + MCP + SDK references, the security docs, and an architecture deep dive — reusing the top-level `docs/` via `{{#include}}` so there is one source of truth; a `just docs` recipe builds it. The **clean-clone quickstart** is verified end to end (build → run → first query) and the README points into the site. A **native TypeScript `DcpeCipher`** closes the last DCPE SDK gap, so Rust, Python, and TypeScript all have native ciphers validated by a cross-language known-answer test. Finally, the two **DCPE Scale-And-Perturb hardening** steps deferred by ADR-0031 ship as a breaking **cipher v2**: a key-derived component **shuffle** (a permutation of the dimensions — an exact L2 isometry, so zero recall cost) and an optional ordering-preserving global affine **normalisation** (a per-dimension shift plus a single scalar scale). The ADR confronts the limit honestly: full per-axis variance whitening is anisotropic, breaks the L2 distance-comparison ordering the untrusted server ranks on, and is therefore incompatible with searchable encryption and not offered. DCPE is experimental, off by default, and stored as ordinary L2 vectors, so the breaking cipher change has no on-disk impact and the `kill -9` crash gate is untouched.

**`v0.16.0` ships** the **retro cockpit** ([ADR-0036](adr/0036-retro-cockpit-design-system.md)) — the terminal experience as the headline. The cockpit wears a coherent **Bronze Quiver** brand (bronze chrome, leather borders, parchment text, a verdigris accent, on oak-black), a **logo** whose **V is a 3-D arrowhead**, and a vocabulary of minimalist retro decorations — framed panels, a database drum icon, a collections table with per-row load bars, a points-trend sparkline, a relationship tree, status badges, and a severity-tagged activity log — so an operator understands the data, its structure, and what is happening at a glance. The view code is decoupled from the live client behind a render-to-buffer API (so the same code drives the cockpit, the tests, and the screenshots), and a workspace-isolated `tools/cockpit-shots` generator renders each screen to a committed PNG (`just tui-shots`) for the README and docs — the screenshots are the *real* render of seeded demo data, so they regenerate in one command and never go stale. The new design code is ~98–100% covered; the live asciinema cast remains an owner / real-terminal artifact.

**`v0.17.0` shipped** the **launch-hardening** pass: a 35× upsert build-time fix (batch WAL sync, [ADR-0038](adr/0038-batch-wal-upsert.md)), one-command install + `quiver update` ([ADR-0039](adr/0039-one-command-install.md)), and the scientific multi-DB benchmark suite ([ADR-0037](adr/0037-scientific-multi-db-benchmark-suite.md)).

**`v0.18.0` ships** the **post-launch correctness, benchmark, and RAG pass**, opened by a full [state-of-Quiver assessment](analysis/state-of-quiver-v0.17.md): **query cost limits** enforced at the op layer, closing an authenticated-DoS and the one docs/code mismatch ([ADR-0040](adr/0040-query-cost-limits.md)); a **deep large-data benchmark** ([ADR-0041](adr/0041-deep-benchmark.md)) with real **SIFT1M** (Quiver second only to FAISS on single-thread QPS at recall ≥ 0.95) and **GIST1M** (Quiver matches FAISS on recall) multi-DB comparisons, a Docker **Milvus-server** adapter, and a fixed server-RSS sampler; **RAG/agentic ergonomics + usage docs** ([ADR-0042](adr/0042-rag-ergonomics.md)) — an async Python client, batched-upsert/scroll/delete-by-filter helpers, a **Haystack** `DocumentStore` (joining the LangChain/LlamaIndex adapters), a `rerank` helper, an MCP `collection_info` introspection tool, and end-to-end RAG / agentic / tuning guides with a runnable example; **hybrid (dense + sparse) search with RRF fusion** ([ADR-0043](adr/0043-hybrid-sparse-search.md)) reachable from the engine, REST, and the Python SDK; and a cockpit that shows the **release version beside the wordmark** plus richer at-a-glance vitals. *Tracked follow-ups (ADR-0043 et al.): gRPC/MCP/TypeScript hybrid parity, a BM25/full-text tokenizer over the sparse path, a derived sparse inverted index, and the GIST1M/Deep10M reference-hardware comparison.* Out-of-scope-for-now, recorded honestly: distributed sharding/consensus, GPU, and lock-free MVCC reads.

**On `develop` (PR #386) — the audit-remediation, hardening, and scale pass** — a full adversarial audit found and fixed **18 confirmed bugs + 1 plausible**, all regression-tested with `just verify` green throughout. Leading the correctness story: a **critical Raft state-machine persistence fix** — applied-state and snapshot were held only in memory, so a voter restart re-applied the entire committed log and a post-compaction cold restart could not recover; state is now persisted on apply/build/install, reloaded on open, with a create-idempotency guard for the crash window. Alongside it, **`next_collection_id` recovery from replayed WAL creates** closes an id-reuse / silent-data-loss window after a crash before checkpoint, and an **IVF-PQ metric-orientation fix** stops it reporting the raw ordering value instead of the true metric (which inverted cosine ranking under MVCC). The remaining fixes span exact large-integer filters (no more `f64` coercion above 2⁵³), cluster-router mutation auditing and an honest `501` on un-routed router ops (instead of silently querying the empty local engine), durable raft-log-dir fsync, atomic coordinator state writes, a fresher migration copy, percent-encoded Python-SDK path segments, and a set of lower-severity correctness/robustness fixes (signed-zero secondary-index filters, DCPE tag normalization binding, IVF tombstone skipping, status-based `404` classification, MCP `k`/`limit` caps + dim validation, `rrf_k0` guard, malformed-sparse rejection, out-of-order-safe OpenAI-embedding indexing). Hardening adds authorize-before-write guards, Admin-gating `GET /cluster/map`, zeroizing the decrypted DEK, binary-search secondary-index ranges, and honest docs (crypto-shred is keyring-dependent, the disk index is rebuildable, BM25 is single-modality). The scale enhancements — each **measured on a modest laptop** (WSL2, ~12 GiB available RAM, 20 cores, single NVMe; dim 128, L2) — are: a **deterministic 262k training sample** for IVF/PQ codebooks in place of the full O(n) scan (1M build **1718s → 115s, ~15×**; small-N byte-identical, all tests unchanged); **nlist ≈ √n** query planning replacing the fixed 64 that forced a full PQ scan (1M query p50 **66ms → 13.8ms, 4.8×**; ~10× at scale); **eliding the redundant L2/Dot build copy** by borrowing via `Cow` instead of normalizing every vector into a second arena (what let 10M fit in a **~13.1 GiB** build peak instead of OOMing at ~18 GiB); and **auto-checkpoint** that seals the active segment at a byte budget (default 256 MiB, `QUIVER_CHECKPOINT_BYTES`) so ingest RSS stays bounded (768 MiB at 1M vs climbing to GiB before). The headline: a **10M-vector index built and served on that laptop** at **query p50 75.8ms / p95 89.6ms** and a frugal **529 B/vector** on disk — against a pre-enhancement 8M run measuring p50 815ms (fixed nlist = full scan), i.e. *more* data at ~10× lower query latency. Honest limits, stated plainly: recall@10 is only measured up to the 2M brute-force cap (IVF-Flat exact = **0.998**, confirming engine coverage; IVF+PQ on the synthetic clustered corpus is a memory-for-accuracy trade, not a defect — the in-tree structured-data PQ suite holds **≥ 0.70**, and no misleading PQ recall number is presented); and the batch index build still materializes all n×dim floats in RAM (~5 GiB at 10M, ~51 GiB at 100M), so on this box the full build tops out around **10M** — 100M is functional-by-design pending the streaming build below and is **never claimed to have been run here**.

**Per-release DoD (`v0.4.0`, `v0.5.0`, …):** the release's features are tested and documented (ADR/README/`.env.example`); coverage ≥ 80%; `just verify` and the SDK suites green; an owner-approved tag via the fast-forward release mechanic.

**Launch DoD (`v1.0.0`, several releases out):** the README benchmark table is complete and reproducible (on documented reference hardware, never fabricated); the quickstart works from a clean clone in minutes; the docs site is live; the cockpit cast is recorded; all tests/property-tests/fuzzers green; history clean and entirely the owner's. Tag **`v1.0.0`** on `main`.

## Future / next

Forward-looking work already scoped and honestly sequenced (not yet gated to a release): a **streaming / chunked index build** — read from the store in chunks, sample-train, stream-encode — as the single-box **100M enabler** that removes the current ceiling (the batch build's resident n×dim float scan, ~5 GiB at 10M and ~51 GiB at 100M); this is an ADR-level change with lock-model implications, deliberately not rushed. Then an **on-disk or sharded primary index** (the `ext_id`→location map is fully resident at ~316 B/point, ~31 GiB at 100M, so 100M today wants a large box); a **binary-quantization frugal-at-scale path**; **WAL-record AAD binding** (an on-disk format change with its own ADR); **concurrent scatter-gather cluster search**; a **per-collection codec cache**; **batched raft entries**; **GPU build/search** (design-only today); and continued **distributed sharding maturity**.

## Testing posture (every phase)

Unit; `proptest` for index/storage invariants; `cargo-fuzz` for wire-protocol + on-disk parsers; crash-recovery tests; `loom` for index concurrency; recall/latency regression gates on a fixed dataset; full server+SDK integration round-trip; `criterion` microbenchmarks for SIMD. Coverage gate starts at 70% and rises to 80%+.
